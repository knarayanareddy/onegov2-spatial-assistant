# GreenPT knowledge chatbot — Phase 1 change set

Apply on top of the scenario-engine build (`govtechnl/onegov2-spatial-assistant`
+ the `onegov2-scenario-wiring` change set). All paths are repo-relative.

Phase 1 = a **grounded, Dutch (B1) knowledge chatbot + curated FAQ registry** that
answers questions about the build, the datasets and the methodology, surfaces the
honest data caveats, and **explains an already-produced ScenarioCard** (read-only).
It reuses the repo's GreenPT `make_llm`, the scenario router's SSE convention, and
the `_llm_metadata_*.json` data dictionary.

## New files — backend
- `src/backend/app/services/chatbot/` — the chatbot package:
  - `models.py` — dataclasses (Citation, Passage, ChatAnswer, FAQEntry).
  - `text.py` — Dutch tokenizer, stopword list, compound **decompounding**
    (`drinkwaterzekerheid` → `drinkwater` + `zekerheid`).
  - `corpus.py` — builds the knowledge corpus from the v3 design doc, the data
    dictionary, the sourced assumptions, the curated FAQs, and **first-class
    data-caveat passages**.
  - `retrieval.py` — Okapi BM25 (pure Python) + idf-weighted coverage as a
    confidence gate.
  - `intents.py` — deterministic router: knowledge / data_limits /
    explain_scenario / scenario_run_request.
  - `answer.py` — B1-Dutch composer: GreenPT (`make_llm`, temperature 0) when a
    key is set, deterministic **extractive fallback** otherwise; always cited.
  - `scenario_context.py` — turns a ScenarioCard into retrievable explain-mode
    passages (cited by scenario hash + stable URL).
  - `faq.py` — 10-entry curated Dutch FAQ registry (incl. a data-limitations FAQ).
  - `service.py` — orchestration; the single entry point `answer_question(...)`.
- `src/backend/app/routers/chatbot.py` — `POST /api/chatbot/ask` (SSE) +
  `GET /api/chatbot/faqs` + `GET /api/chatbot/faqs/{id}`.
- `src/backend/tests_chatbot/` — 29 offline tests (corpus, retrieval, intents,
  answer fallback + injected-fake-LLM, explain-scenario, FAQ, SSE endpoint).

## New files — frontend (drop-in; build in your environment)
- `src/frontend/src/composables/useChatbotSSE.ts` — consumes the chat SSE stream
  (mirrors `useScenarioSSE.ts`), plus a `fetchFaqs()` helper.
- `src/frontend/src/components/chatbot/ChatbotPanel.vue` — the chat tab: streaming
  answer, per-answer **Bronnen** list, *veelgestelde vragen*, a clarifying-question
  chip, and an optional `scenarioCard` prop for explain-mode.

## Modified files — 2 small, additive edits
- `src/backend/app/main.py` — **2 lines**: add `chatbot,` to the
  `from app.routers import (...)` tuple, and `app.include_router(chatbot.router)`
  after `app.include_router(scenario.router)`. (If you prefer not to overwrite
  your `main.py`, just make those two edits by hand.)
- `src/frontend/src/App.vue` — additive: a `💧 Kennis-chat` toggle button beside
  the `🔮 Scenario-engine` toggle (mutually exclusive), rendering `ChatbotPanel`.

## Behavior notes
- **GreenPT key handling.** `GREENPT_KEY` lives in `src/backend/.env` (never in
  chat). When it is set, answers are generated by `make_llm` at **temperature 0**;
  otherwise a deterministic extractive Dutch fallback runs. The GreenPT path is
  unit-tested via an injected fake LLM, so tests stay offline.
- **Retrieval.** Keyword BM25 with Dutch stopwords + compound splitting. GreenPT
  *does* expose `/v1/embeddings` (verified: returns 401, i.e. the path exists and
  needs a key), so an embeddings-backed retriever is a clean drop-in upgrade — not
  needed for this small corpus and it would break offline determinism.
- **Hard safety boundary (preserved).** Phase 1 is **read-only**. It never writes
  or executes code/SQL and never runs a scenario. A request to *compute* a new
  scenario is **routed** to the Scenario-engine with a grounded explanation of the
  levers — it is not executed. No dead ends: low-confidence retrieval returns a
  clarifying question.
- **Honest data caveats are first-class.** The corpus indexes the design doc's
  caveats and a dedicated `data_limits` intent foregrounds them: verzilting
  `ZOUT_CONC` is a saturated single class (a mask); CBS consumption is a relative
  density proxy (~46k), not headcounts; `drinkwater_productieketen` and
  `toestandsbeoordeling_oppervlaktewaterlichamen` ship empty.

# Phase 2 — scenario-from-chat

Let a user describe a scenario in the Kennis-chat and get a real ScenarioCard +
map (and optional with/without-shock comparison) back. The engine already runs a
free-text question end-to-end, so Phase 2's new, safety-critical contribution is
a **validation/whitelist gate** between extraction and execution.

## New files
- `src/backend/app/services/chatbot/scenario_run.py` — the gate:
  `prepare_scenario_request()` (extract → validate against the whitelist → detect
  comparison → runnable plan **or** a clarifying question) and `execute_plan()`
  (runs the existing `run_scenario` / `run_comparison` with the pre-validated
  params). Whitelist: allowed scenario types, KNMI presets (B/Hd/Hn/Ld/Ln), growth
  presets, a fixed set of assumption-override **lever keys**, known intakes,
  horizon 2025–2100, and a minimum confidence. Anything else → clarify, never run.
- `src/backend/tests_chatbot/test_scenario_run.py` — 11 tests (whitelist accept/
  reject, unknown lever, unknown intake, missing location, low confidence,
  comparison detection, real single + comparison runs, SSE endpoint run + clarify).

## Modified files
- `src/backend/app/routers/chatbot.py` — `POST /api/chatbot/ask` now branches:
  a `scenario_run_request` intent is validated and (if runnable) executed via the
  engine, streaming `scenario_params_confirmed` then the same scenario events as
  the engine (`reasoning_step`, `feasibility_class`, `scenario_card`, `map_data`,
  `scenario_delta`, `done`). Cards are persisted to the shared `ScenarioStore`, so
  the stable URL and explain-mode work on chat-run scenarios too. Knowledge /
  data_limits / explain answers keep their Phase 1 events.
- `src/backend/app/services/scenario/workflow.py` — **backward-compatible**:
  `run_scenario` / `run_comparison` gain an optional `params=` argument; when given,
  `extract_node` uses those pre-validated params verbatim (no double extraction).
  Default behaviour (and all 27 scenario tests) unchanged.
- `src/frontend/src/composables/useChatbotSSE.ts` — dispatches the scenario events.
- `src/frontend/src/components/chatbot/ChatbotPanel.vue` — renders a scenario
  result (verdict badge, score, reasoning steps, sources, comparison delta) with an
  **Open-in-Scenario-engine** handoff via the stable URL (reuses the Deck.gl map
  rather than duplicating it).

## Safety (unchanged thesis)
The LLM only maps language → **validated** ScenarioParams + whitelisted assumption
overrides. The deterministic engine validates and runs. No code/SQL synthesis. An
unknown lever, unrecognised intake, or low confidence yields a clarifying question
and is **never** executed. Verified by `test_scenario_run.py` (e.g.
`test_unknown_lever_is_rejected`).

# Phase 3 — FAQ caching + ranking

Confident, grounded chat answers become a growing FAQ: cached as moderation
*suggestions* (PII anonymised, sourced, version-stamped), the most-asked
*published* ones surface as veelgestelde vragen ranked by hits, and the cache
invalidates on data/doc/git drift — the same trust mechanism as the scenario cache.

## New files
- `src/backend/app/services/chatbot/pii.py` — anonymise Dutch postcodes, e-mail,
  phone before storage.
- `src/backend/app/services/chatbot/faq_cache.py` — the cache core: `FaqCacheEntry`,
  question normalisation, keyword `cache_key` (PII stripped, paraphrase-collapsing),
  `version_stamp` = hash(git_commit + corpus signature) + stale filtering,
  `should_cache` (grounded+confident gate), `capture_answer`, `published_as_faqs`,
  and the **in-memory store** (default, offline-tested) behind a small async interface.
- `src/backend/app/models/faq_cache.py` — SQLModel `FaqCache` table (shared Postgres).
- `src/backend/app/services/chatbot/faq_cache_sql.py` — `SqlFaqCache`, the Postgres
  adapter (same async interface). Enable with `set_store(SqlFaqCache(engine))` at startup.
- `src/backend/alembic/versions/b2f3a1c4d5e6_add_faq_cache_table.py` — the migration.
- `src/backend/tests_chatbot/test_pii.py`, `test_faq_cache.py` — 18 tests.

## Modified files
- `src/backend/app/routers/chatbot.py` — after a confident, cited knowledge/
  data_limits answer, best-effort `capture_answer` (never breaks the response).
  `GET /api/chatbot/faqs` now returns curated FAQs + published cache ranked by hits.
  New auth-gated moderation routes: `GET /faqs/suggested`, `POST /faqs/suggested/{id}/promote`,
  `POST /faqs/suggested/{id}/reject` (promote requires the grounded-with-sources re-check;
  a rejected entry can't be promoted).
- `src/backend/alembic/env.py` — import the new model so autogenerate sees it.

## Storage choice (important)
The sandbox cannot run Postgres (DB wire protocols are firewalled), so the
**in-memory store is the tested default** and a graceful fallback when Postgres
isn't configured. For durable, shared storage use the Postgres adapter:
1. `alembic upgrade head` (creates `faq_cache`).
2. At startup: `from app.database import engine; from app.services.chatbot.faq_cache import set_store; from app.services.chatbot.faq_cache_sql import SqlFaqCache; set_store(SqlFaqCache(engine))`.
The cache LOGIC is identical and fully tested via the in-memory store; validate the
Postgres adapter against your database.

## Moderation policy (brief open Q2)
Auto-cache as *suggested* only. Promotion to *published* runs through an auth-gated
endpoint and requires the grounded-with-sources re-check. Repo auth is a single-user
stub (`get_current_user`), so the logged-in user reviews; restrict to an admin role
in your env. New `requirements`: `sqlmodel` (already a repo dependency).

# Phase 4 — declarative recipe-builder + Postgres activation

## Recipe-builder (data, not code)
A recipe is a validated, declarative combination of the H3 composite signal
weights + presets — never code/SQL. New files/edits:
- `src/backend/app/services/chatbot/recipe.py` — `recipe_schema()`,
  `validate_recipe()` (weights must sum to 1 within 0.01 or it's rejected with a
  clarifying message; whitelisted signals/presets/ranges; known intakes), and
  `execute_recipe()` (maps to ScenarioParams + assumption_overrides and runs the
  existing engine).
- `src/backend/app/services/scenario/workflow.py` — **backward-compatible**:
  `run_scenario` gains an optional `base=` ("salinity" | "populated") so a recipe
  can pick the scoring universe; default keeps the housing-aware behaviour.
- `src/backend/app/routers/chatbot.py` — `GET /api/chatbot/recipe/schema` and
  `POST /api/chatbot/recipe/run` (SSE: `scenario_params_confirmed` → the engine
  ScenarioCard; a validation miss streams a clarifying `followup_question`).
- Frontend: `ChatbotPanel.vue` gains a collapsible recipe-builder (four weight
  sliders with a live sum-to-1 guard, KNMI/growth/base selectors, optional
  location) that renders the resulting ScenarioCard; `useChatbotSSE.ts` gains
  `runRecipe` + `fetchRecipeSchema`.
- `tests_chatbot/test_recipe.py` — 13 tests (validation, mapping, real single
  run, populated-universe switch, schema + run endpoints, clarify on bad weights).

## Postgres FAQ-cache activation (Phase 3 store, now wired)
- `src/backend/app/config.py` — new `FAQ_CACHE_BACKEND` ("memory" default | "postgres").
- `src/backend/app/main.py` — lifespan calls `set_store(SqlFaqCache(engine))` when
  `FAQ_CACHE_BACKEND=postgres` (guarded; falls back to in-memory on error).
- `tests_chatbot/test_faq_cache_sql.py` — exercises `SqlFaqCache` against SQLite
  (skipped if `aiosqlite` is absent).

### To turn on Postgres persistence in your environment
```bash
cd src/backend
export FAQ_CACHE_BACKEND=postgres            # and your DATABASE_URL (postgresql+asyncpg://…)
alembic upgrade head                         # creates the faq_cache table
# restart the app — suggestions/published FAQs now persist in Postgres
```
The sandbox firewalls the Postgres wire protocol, so this was **verified against a
real SQL engine (SQLite via aiosqlite)** end-to-end instead: `alembic upgrade head`
created `faq_cache` (12 columns), the `SqlFaqCache` adapter passed dedup/hits,
promote, published-vs-stale, reject, promote-after-reject, and bad-id guards, and
`alembic downgrade -1` dropped the table cleanly. The model deliberately uses
portable column types (TEXT for JSON), so the SQLite run is representative; run the
same `alembic upgrade head` against your Postgres.

# Phase 6 — accountability (auth + audit) + uncertainty band

Closes the two serious gaps from the speculative critique.

## Gap A — real auth + audit trail (Woo/accountability)
- `app/auth.py` is now **AUTH_MODE-aware**: `dev` (single local user, default — unchanged),
  `jwt` (verify a Bearer token: HS256 secret or RS256 via `AUTH_JWT_JWKS_URL`), or `header`
  (trust an OIDC reverse-proxy header). Adds `roles` to `CurrentUser`, a `require_role`
  dependency (FAQ moderation is now **admin-only**), and an `AUTH_REQUIRED` gate. New
  settings in `config.py` (`AUTH_MODE`, `AUTH_JWT_SECRET`, `AUTH_JWT_ALGORITHM`,
  `AUTH_JWT_JWKS_URL`, `AUTH_JWT_AUDIENCE`, `AUTH_HEADER_*`, `AUTH_ADMIN_ROLE`). New dep: **PyJWT**.
- **Audit trail:** `app/services/audit.py` (`AuditEntry` + in-memory store + `record_audit`),
  `app/models/audit.py` (`AuditLog` SQLModel), `app/services/audit_sql.py` (Postgres adapter),
  + Alembic migration `c3d4e5f6a7b8`. `GET /api/audit` (auth-gated). Scenario run / cumulative /
  verify / uncertainty + recipe + FAQ promote/reject all write **who-did-what** entries.
- **Identity in the citation:** the running user is stamped into the stored card (`run_by`) and
  into `build_citation` ("Uitgevoerd door: …").
- Honest constraint: no live IdP in the sandbox, so the jwt path is verified with a **minted
  test token**; point `AUTH_JWT_*` at your Keycloak/Azure AD in your env. Default (`dev`) is
  unchanged. Postgres audit table is SQLite-verified; run `alembic upgrade head` + set
  `AUDIT_BACKEND=postgres`.

## Gap B — uncertainty band on the verdict (false-precision fix)
- `app/services/scenario/uncertainty.py` + `POST /api/scenario/uncertainty`: sweeps the five
  KNMI'23 presets (B/Ln/Hn/Ld/Hd) and returns the **score band**, the **verdict distribution**
  ("STOP in N/5"), a **robust vs knife-edge** flag, and a Dutch narrative. Deterministic.
- Frontend: a "Robuustheid (KNMI-band)" button + a robustness line on the `ScenarioPanel` verdict.

# Phase 7 — assumption versioning (Gap F) + calibration & validation status (Gap E)

## Gap F — assumption-library versioning ("reproduce as advised")
- `app/services/scenario/assumptions.py`: `ASSUMPTIONS_VERSION` + a structured
  `ASSUMPTIONS_CHANGELOG` + `assumption_library()` (the sourced weights/thresholds +
  KNMI/growth maps + version). `GET /api/assumptions` exposes it.
- Every ScenarioCard now carries `assumptions_version` (set in `format_node`), stamped into
  `build_citation` ("Aannameversie: vX") and persisted; `/verify` reports **assumption-version
  drift** (cached vs current) alongside dataset drift — so "reproduce exactly as advised on
  [date]" is answerable.

## Gap E — calibration + honest validation status
- `app/services/scenario/calibration.py`: `CALIBRATION_CASES` + `run_calibration()` (agreement
  report) + `validation_status()`. `GET /api/scenario/calibration` runs the harness.
- The card + citation carry a `validation_status` string. **Honest framing:** no external
  capacity study ships, so the calibration set is the design's own verified reference figures
  (IJssel KNMI B→CAUTION ~50, Hd→STOP ~82 — both reproduce exactly, 2/2 agreement) — a
  *reproducibility/consistency* calibration. The status says explicitly it is **not** validated
  against an external study, and `CALIBRATION_CASES` has a pluggable slot (`kind="external-study"`)
  for real Dunea/Evides cases. See `docs/CALIBRATION.md`.
- Frontend: a "Herkomst & validatie" line (aannameversie + validatiestatus) on the ScenarioPanel.
